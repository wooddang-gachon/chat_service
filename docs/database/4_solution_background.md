# 데이터베이스 설계 및 성능 최적화 지식 (Solution Knowledge)

이 문서는 대용량 데이터베이스 테이블의 열 개수 최적화 가이드와 SQL 및 Real MySQL 기반의 데이터베이스 물리 구조 설계 지식을 통합하여 정리한 문서입니다.

---

## 1. 테이블의 열(Column) 개수와 성능 최적화

### 1.1 테이블의 열이 많아질 때 발생하는 성능 저하 원인

- 디스크 I/O 및 페이지(Page) 효율성 저하: 관계형 데이터베이스(MySQL InnoDB 등)는 데이터를 디스크에서 메모리로 읽어올 때 페이지(기본 16KB) 단위로 처리합니다. 테이블의 열이 많아 한 행의 크기가 커지면, 단일 페이지 내에 적재할 수 있는 데이터 행의 개수가 줄어듭니다. 이 때문에 동일한 데이터 건수를 조회(Scan)하더라도 훨씬 더 많은 수의 디스크 페이지를 로드해야 하므로 물리적 디스크 I/O 부하가 급증합니다.
- 메모리 버퍼 풀(Buffer Pool) 캐시 효율성 급감: 자주 사용되는 데이터 페이지를 메모리에 보관하는 버퍼 캐시 공간은 한정되어 있습니다. 각 페이지당 행의 밀도가 낮으면 캐시 메모리 공간이 비효율적으로 낭비되어 캐시 적중률(Cache Hit Ratio)이 떨어집니다. 이로 인해 메모리 조회가 실패하고 디스크를 반복해서 읽는 병목 현상이 심화됩니다.
- 로우 오버플로우 (Row Overflow) 현상: MySQL InnoDB는 한 행의 전체 데이터 크기가 페이지의 약 절반(약 8KB)을 초과하면, 가변 길이 데이터(VARCHAR, TEXT 등)를 해당 행에 직접 담지 않고 외부 오버플로우 페이지(Overflow Page)로 격리시킵니다. 해당 행에는 오버플로우 페이지를 가리키는 20바이트 주소 포인터만 유지되므로, 해당 열을 조회할 때마다 디스크 포인터를 따라 다른 페이지를 추가로 읽어야 하는 다중 디스크 조회 부하가 발생합니다.
- 무의식적인 SELECT _ 전송 비용 증가: 열의 개수가 수십 개가 넘어가는 테이블에 SELECT _ 쿼리를 전송하면, 데이터베이스 서버뿐만 아니라 네트워크 전송 대역폭(Bandwidth) 낭비와 클라이언트/애플리케이션 메모리 버퍼에 불필요한 과부하를 가하게 됩니다.

### 1.2 해결 방안: 수직 분할 (Vertical Partitioning)

가로로 비대한 테이블을 자주 조회하는 핵심 열 그룹과 부가적인/용량이 큰 열 그룹으로 분리하는 것입니다.

- 핵심 정보와 상세 정보 분리: 단건 단기 조회용 필드(아이디, 상태코드 등)는 가볍고 빠르게 조인될 수 있도록 날씬한 슈퍼타입 테이블에 두고, 본문이나 상세 설명 등은 별도 서브타입 테이블로 격리합니다.
- 상속 모델링(Subclassing) 활용: 카테고리나 유형에 따라 쓰이는 속성들이 상상이 다를 때 이를 1:1 관계의 하위 서브 테이블로 나누어 스펙에 맞게 NOT NULL 제약을 걸고 메인 테이블은 작게 유지합니다.

## 2. 물리적 저장 구조 및 인덱스 설계 지식

### 2.1 B-Tree 인덱스

- 인덱스는 테이블의 쓰기 성능을 일부 희생하고 읽기 성능을 극대화하기 위해 B-Tree 구조로 구성됩니다.
- 인덱스 키의 크기가 커지면 B-Tree 자식 노드의 개수가 줄어들어 인덱스 탐색 깊이가 깊어지고, 메모리 캐싱 효율이 저하됩니다. 따라서 인덱스에 포함될 컬럼은 최소한으로 유지해야 합니다.

### 2.2 기본키(PK) 구성이 미치는 영향

- RDB(특히 MySQL InnoDB)에서 INSERT할 때 데이터는 PK(기본키)를 기준으로 물리적으로 정렬되어 삽입됩니다(클러스터형 인덱스).
- PK 구성이 불규칙하거나 무작위(예: UUID v4 등)이면 새로운 행이 삽입될 때 기존 페이지 사이에 데이터를 끼워 넣어야 하므로, 페이지 분할(Page Split)이 빈번하게 발생하여 디스크 I/O 성능이 극적으로 저하됩니다.
- 따라서 PK는 가급적 순차적으로 증가하는 값(Auto Increment)이나 시간 기반의 정렬 가능한 값을 활용하는 것이 쓰기 성능 확보에 유리합니다.

## 3. 실제 스키마 개선 사례 분석

### 3.1 MESSAGES 테이블 상속 모델링

- 개선 전: MESSAGES 테이블에 content(본문 텍스트), S3 파일 상세 정보, 이모티콘 맵핑 코드 등이 한데 모여 있어 텍스트 전송 시 많은 컬럼들이 NULL로 채워졌습니다.
- 개선 후: 공통 메타데이터(id, type, created_at, client_message_token 등)만 MESSAGES 테이블로 들고 가고, 본문 내용은 TEXT_MESSAGES에, 파일 정보는 MEDIA_MESSAGES로 물리적으로 수직 분할했습니다.
- 기대 효과: 대화방의 대량 메시지 렌더링 시 무거운 파일 정보 주입 없이 텍스트 중심의 메시지들만 가볍고 빠르게 조회할 수 있습니다.

### 3.2 PROFILES와 PROFILE_ASSETS 분리

- 개선 전: PROFILES 테이블에 최신 프로필 사진 및 과거 프로필 이력 정보까지 중구난방으로 관리하려 했습니다.
- 개선 후: 현재 쓰이는 최신 프로필 이미지만 PROFILES 테이블에 캐싱해두고, 과거 사진첩 히스토리 조회는 독립된 PROFILE_ASSETS 테이블로 이력을 오프로딩했습니다.
- 기대 효과: 친구 목록 로딩 등 최신 프로필 정보만 대량으로 매핑해야 할 때 불필요한 과거 이력 데이터 스캔 없이 즉각 조회가 가능합니다.

## 4. 개발 가이드라인 및 베스트 프랙티스

- 필요한 컬럼만 명시하여 조회: 쿼리 작성 시 가급적 SELECT \* 사용을 지양하고, SELECT id, nickname 과 같이 필수 필요한 컬럼만을 엄격하게 지정하여 호출합니다.
- VARCHAR 최대 크기의 합리적 지정: 사용하지도 않는 varchar(4000) 등을 남발하여 선언하기보다는, 적절한 길이(예: varchar(50))를 부여하여 예측 가능한 메모리 구조를 가질 수 있도록 통제해야 합니다.
- 적절한 인덱스(B-Tree) 설계: 복합 인덱스 구성 시, 선두 컬럼에는 항상 카디널리티(변별력)가 높은 핵심 필수 식별자를 선언하고 쿼리에 자주 등장하는 경량 컬럼 위주로 태워 인덱스 파일 크기 자체도 가볍게 유지합니다.

## 5. BigQuery 쿼리 성능 최적화 가이드

이 영역은 Google Cloud BigQuery 공식 문서를 바탕으로 대용량 데이터 쿼리 최적화 및 비용 절감을 위한 핵심 아키텍처와 모범 사례를 정리합니다.

### 5.1 쿼리 성능을 평가하는 5대 핵심 요소

- 입력 데이터 및 데이터 소스 (I/O): 쿼리 실행 과정에서 디스크 또는 외부 데이터 소스로부터 읽어 들이는 바이트 수입니다.
- 노드 간 통신 (셔플, Shuffle): 쿼리 실행의 중간 단계에서 슬롯(작업 노드) 간에 주고받는 데이터의 양입니다.
- 계산 (Computation): 쿼리 연산(함수 처리, 정렬, 조인 등)에 필요한 CPU 작업량입니다.
- 출력 (구체화, Materialization): 쿼리 결과를 테이블에 쓰는 등의 출력 바이트 수입니다.
- 용량 및 동시 실행 (Capacity & Concurrency): 사용 가능한 슬롯(Slot) 용량 및 동시에 실행 중인 작업의 개수입니다.

### 5.2 용량 및 동시 실행 최적화

- 가격 모델 및 슬롯: BigQuery는 주문형(On-demand) 모델과 전용 슬롯을 할당받는 용량 기반(Capacity-based) 모델을 지원합니다.
- 슬롯 자동 확장 (Slots Autoscaling): 워크로드 부하에 따라 동적으로 슬롯을 늘리거나 줄여 리소스 병목을 예방할 수 있습니다.
- 공정 예약 (Fair Scheduling): BigQuery는 특정 쿼리가 자원을 독점하지 않도록 여러 실행 중인 쿼리 간에 자원을 공정하게 분배합니다. 슬롯이 부족한 쿼리는 쿼리 큐(Query Queue)에서 대기하게 됩니다.

### 5.3 쿼리 계획(Query Plan) 및 실행 그래프 분석

- 총 슬롯 시간(Slot Time, 초): 쿼리가 사용한 총 CPU 시간의 합산입니다. 슬롯 시간이 낮을수록 효율적인 쿼리이며 비용이 절감됩니다.
- 셔플 스테이지 최적화: 입력 데이터 행 수보다 조인 후의 출력 행 수가 급격하게 많아지는 JOIN 스테이지 등이 감지되면, 쿼리의 앞 단계에서 미리 필터링(WHERE 절 적용 등)을 수행하여 셔플 데이터양을 최대한 줄여야 합니다.
- BI Engine 활용: 자주 사용되는 데이터를 메모리에 캐싱하여 쿼리를 가속화해 주는 분석 엔진으로, 코드 수정 없이도 성능 향상이 가능합니다.

- 출처: [Google Cloud BigQuery 공식 문서 - 쿼리 성능 최적화 소개](https://docs.cloud.google.com/bigquery/docs/best-practices-performance-overview?hl=ko)

## 6. 관계형 데이터베이스(RDB) 설계 고민과 유연한 아키텍처 (인하대 후문 프로젝트 사례 중심)

이 영역은 실제 서비스 개발 과정에서 발생하는 RDB 설계 고민과 확장성, 유연성을 고려한 모범 사례를 정리합니다.

### 6.1 Soft Delete vs Hard Delete 의 선택 기준

- 히스토리(History) 데이터로서의 가치 분석: 단순히 즐겨찾기(북마크) 취소와 같은 삭제 이벤트라도, 사용자의 과거 선호도 흐름이나 트렌드 통계 분석을 위한 유의미한 원천 데이터가 될 수 있으므로 Soft Delete 방식을 우선 고려합니다.
- 물리적 비용 대비 효용 분석: 타임스탬프 저장은 8바이트에 불과하여 약 1억 건의 데이터가 모여야 1GB를 차지하므로, 미미한 물리적 용량 절약을 위해 비즈니스 히스토리 가치를 상실하는 Hard Delete를 성급히 선택할 필요는 없습니다.
- 단순 관계 매핑 테이블: 부가적인 정보 없이 두 엔터티의 식별자만을 단순 연결하는 매핑 테이블(예: place_type_mapping)은 Soft Delete 보다는 Hard Delete를 사용하는 것이 스키마를 단순하게 관리하는 데 유리합니다. 단, 연결되는 원본 엔터티(place, place_type 등)에는 Soft Delete를 적용합니다.

### 6.2 데이터 타입 설계의 유연성 (VARCHAR vs VARCHAR(N))

- 닉네임 등 특정 컬럼의 길이를 10글자로 제한하더라도, DB 레벨에서 VARCHAR(10)과 같이 엄격한 제약을 주기보다 VARCHAR(또는 넉넉한 크기)로 설계하고 실제 유효성 검증은 애플리케이션(코드) 단에서 처리하는 것이 유연성을 확보하기에 좋습니다.
- 이 방식은 향후 기획 사양 변경(예: 글자 수 제한을 12글자로 상향) 시 DB 스키마 변경 및 마이그레이션 부담 없이 비즈니스 로직 수정만으로 대처할 수 있게 합니다.

### 6.3 테이블 네이밍의 의미론적 접근

- N:M 관계의 매핑 테이블을 생성할 때, 기계적으로 tableA_tableB_mapping과 같이 짓는 것은 직관성을 떨어뜨릴 수 있습니다.
- 대신 비즈니스 도메인의 결과를 나타낼 수 있는 명확한 명칭(예: 서비스 설문 매핑 테이블을 service_result로 명명)을 선택하여 의미를 명확히 하고, 구체화하기 곤란할 때에만 예외적으로 mapping 접미사를 사용하는 방식으로 유연성을 확보합니다.

## 7. RDB 데이터베이스 설계의 반복적 접근 및 정규화 원칙

이 영역은 RDB 논리 설계 단계에서 데이터 무결성을 지키고 변경에 강한 데이터베이스를 설계하기 위한 점진적 프로세스와 아키텍처 원칙을 정의합니다.

### 7.1 점진적이고 반복적인 설계 프로세스

- 초기 설계의 부담 완화: 처음부터 완벽한 ERD를 그리려고 많은 시간을 소모하기보다, 핵심 테이블과 대략적인 컬럼 구조를 빠르게 정의한 후 전체를 조감하며 지속적으로 피드백하고 추가/수정하는 이터레이티브(Iterative) 접근 방식을 취합니다.
- 관심사의 분리 (작은 그룹화): 거대한 시스템을 한꺼번에 설계하는 대신, 비즈니스 단위(예: 상품-카테고리 관계, 상품-주문 관계 등)별로 작은 그룹을 지어 관계를 정리하고, 최종 단계에서 전체 테이블을 연결하여 ERD를 완성합니다.

### 7.2 정규화와 비정규화의 트레이드오프(Trade-off)

- 정규화(Normalization)의 이점: 중복 데이터를 배제하여 데이터 갱신 시 대상 레코드를 최소화할 수 있습니다. 예를 들어, 카테고리명을 수정할 때 정규화된 마스터 테이블 내 단 한 개의 레코드만 수정하면 되므로 데이터 무결성을 쉽게 보장하고 일관된 마스터 데이터 관리가 가능합니다.
- 비정규화(Denormalization)의 활용: 데이터의 갱신/삭제가 거의 일어나지 않으면서 검색(Read)의 성능 극대화가 필요한 특정 컬럼들의 경우에는, 복잡한 JOIN을 회피하기 위해 테이블 내에 고의로 중복 데이터를 허용(비정규화)하여 쿼리 처리 속도를 최적화하는 전략을 세웁니다.

### 7.3 보이지 않는 요구사항의 선제적 설계

- 논리적 삭제 및 이력 보존: 사양서에 명시되어 있지 않더라도, 주문된 상품 정보처럼 과거 정합성 보존이 필수적인 마스터 테이블의 데이터를 물리 삭제(Hard Delete)할 시 연관 비즈니스 데이터가 훼손됩니다. 이에 따라 Soft Delete 플래그 및 등록일/등록자 등 감사(Audit) 정보를 설계 단계에서 선제적으로 추가하여 이력 추적성을 확보해야 합니다.
- N:M 관계로의 미래 확장성 고려: 현재 비즈니스 상 1:N 관계일지라도 향후 다대다(N:M) 관계로의 사양 확장이 빈번하게 예상되는 도메인(예: 상품-카테고리 관계)의 경우, 사전에 중간 관계 매핑 테이블을 생성해둠으로써 미래의 대대적인 재설계 비용을 최소화합니다.

## 8. 데이터 모델링 단계별 프로세스 및 ERD 설계 표준

이 영역은 논리적/물리적 데이터베이스 설계의 골격을 구축하기 위한 체계적인 데이터 모델링 수행 절차와 ERD 표기법 표준을 정의합니다.

### 8.1 데이터 모델링의 4대 진행 절차

- 1단계: 업무 파악 (요구사항 수집 및 분석): 구축 대상 비즈니스 UI와 흐름을 사용자(의뢰인) 관점에서 파악하고 요구사항을 엄격하게 수집합니다.
- 2단계: 개념적 데이터 모델링: 도메인 내 핵심 엔티티(Entity)를 도출하고 피터 첸 표기법 등을 통해 추상적인 개체 간의 유기적 관계(Relationship)를 ERD 형태로 구상합니다.
- 3단계: 논리적 데이터 모델링: 비즈니스 규칙에 근거하여 상세 속성(Attribute), 도메인(Domain) 타입을 명시하고 기본키(PK)와 외래키(FK)를 지정하며, 정보 공학 표기법(IE 표기법, 까마귀발 표기법 등)으로 테이블 구조를 매핑하고 정규화를 실행합니다.
- 4단계: 물리적 데이터 모델링: 최종적으로 활용할 DBMS(예: MySQL 등)의 성능적 특성(인덱스, 파티셔닝 등)을 적용하고 DDL SQL 구문을 작성하여 물리 데이터베이스 테이블을 구축합니다.

### 8.2 엔티티(Entity) 및 속성 분류

- 유형 엔티티: 물리적으로 실재하는 대상입니다 (예: 회원, 상품).
- 무형 엔티티: 형태 없이 비즈념 개념으로 존재합니다 (예: 장바구니, 부서 정보).
- 이력 엔티티: 시간에 따른 행위 변화를 저장하는 테이블입니다 (예: 로그인 기록, 입출고 이력).
- 문서 엔티티: 업무 프로세스상의 전표나 문서를 표현합니다 (예: 주문서, 결제 영수증).
- 코드 엔티티: 공통 기준 정보를 관리합니다 (예: 국가코드, 카테고리 코드).

### 8.3 식별자 관계와 비식별자 관계의 설계 표준

- 식별자 관계 (실선 표시): 부모 엔티티의 기본키(PK)가 자식 엔티티의 기본키(PK) 일부로 포함되는 강한 종속적 관계입니다. 부모 레코드의 생명주기에 자식 레코드가 완전히 종속됩니다.
- 비식별자 관계 (점선 표시): 부모 엔티티의 기본키(PK)가 자식 엔티티의 주 식별자가 아닌 일반 속성(일반 필드)으로 참조(FK)되는 약한 관계입니다. 자식 엔티티가 부모 없이도 독립적으로 고유한 식별 키를 가질 수 있어, 테이블 간 결합도를 낮추는 데 보편적으로 활용됩니다.

## 9. 슈퍼타입/서브타입(Supertype/Subtype) 모델링 기법

이 영역은 객체지향의 상속 관계 구조를 관계형 데이터베이스(RDB)에 물리적으로 구성하기 위한 3대 설계 전략과 선택 기준을 정의합니다.

### 9.1 싱글 테이블 전략 (Single Table, Roll-up)

- 물리적 구조: 슈퍼타입과 서브타입을 하나의 통합 테이블로 병합하여 관리하는 방식입니다. 행의 종류를 구별하기 위한 구분 컬럼(예: type, discriminator)을 필수 배치합니다.
- 장점: 조인(JOIN) 연산이 필요 없어 데이터 조회 속도가 극대화되고 쿼리가 간결해집니다.
- 단점: 특정 서브타입에만 필요한 속성들에 NULL 입력을 허용해야 하므로 컬럼의 NOT NULL 제약조건을 강제할 수 없으며, 서브타입 종류가 늘어남에 따라 디스크 및 버퍼 캐시 효율이 저하됩니다.

### 9.2 개별 테이블 전략 (Table-per-Class, Roll-down, 1:1 분리)

- 물리적 구조: 공통 메타데이터는 슈퍼타입 테이블에 보존하고, 개별 서브타입의 고유 속성은 별도 자식 테이블로 1:1 수직 분할하여 식별자 관계로 연결하는 방식입니다.
- 장점: 자식 테이블 내 고유 컬럼들에 대해 엄격하게 NOT NULL 제약조건을 부여할 수 있어 데이터 무결성이 매우 우수하며, 디스크 페이지 당 행 밀도를 최대로 제어하여 메모리 버퍼 효율을 향상시킵니다.
- 단점: 상세 정보를 조회하기 위해 상시 JOIN 연산이 수반되므로 성능적 오버헤드가 발생할 수 있습니다.

### 9.3 서브타입 단독 테이블 전략 (Table-per-Concrete-Class, Union)

- 물리적 구조: 공통 슈퍼타입 테이블을 생성하지 않고, 자식 서브타입별로 각각의 테이블에 공통 속성과 자식 고유 속성을 모두 독립적으로 기술하는 구조입니다.
- 장점: 특정 서브타입 데이터만 타겟팅하여 단건 조회를 실행할 때 슈퍼타입 테이블과의 조인이 배제되므로 조작 성능이 가장 우수합니다.
- 단점: 자식 테이블 전체에 대해 공통 속성 기준으로 일괄 조회 시 복잡한 UNION 연산이 필요하여 대용량 처리 시 성능이 크게 저하되며, 공통 속성 스펙 변경 시 유지보수 범위가 모든 서브 테이블로 확장됩니다.
