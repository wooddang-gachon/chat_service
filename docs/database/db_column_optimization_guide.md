# 대용량 데이터베이스 테이블의 열(Column) 개수와 성능 최적화 가이드

이 문서는 테이블의 가로 크기(열의 개수 및 레코드의 바이트 용량)가 시스템 성능에 미치는 물리적 영향과 이를 해결하기 위한 설계 기법을 정리합니다.

---

## 1. 테이블의 열이 많아질 때 발생하는 성능 저하 원인

### 1.1 디스크 I/O 및 페이지(Page) 효율성 저하
- 관계형 데이터베이스(MySQL InnoDB 등)는 데이터를 디스크에서 메모리로 읽어올 때 행 단위가 아닌 페이지(기본 16KB) 단위로 처리합니다.
- 테이블의 열이 많아 한 행의 크기가 커지면, 단일 16KB 페이지 내에 적재할 수 있는 데이터 행의 개수가 줄어듭니다.
- 동일한 데이터 건수를 조회(Scan)하더라도 훨씬 더 많은 수의 디스크 페이지를 로드해야 하므로 물리적 디스크 I/O 부하가 급증합니다.

### 1.2 메모리 버퍼 풀(Buffer Pool) 캐시 효율성 급감
- 자주 사용되는 데이터 페이지를 메모리에 보관하는 버퍼 캐시 공간은 한정되어 있습니다.
- 각 페이지당 행의 밀도가 낮으면 캐시 메모리 공간이 비효율적으로 낭비되어 캐시 적중률(Cache Hit Ratio)이 떨어집니다. 이로 인해 메모리 조회가 실패하고 디스크를 반복해서 읽는 병목 현상이 심화됩니다.

### 1.3 로우 오버플로우 (Row Overflow) 현상
- MySQL InnoDB는 한 행의 전체 데이터 크기가 페이지의 약 절반(약 8KB)을 초과하면, 가변 길이 데이터(VARCHAR, TEXT 등)를 해당 행에 직접 담지 않고 외부 오버플로우 페이지(Overflow Page)로 격리시킵니다.
- 해당 행에는 오버플로우 페이지를 가리키는 20바이트 주소 포인터만 유지되므로, 해당 열을 조회할 때마다 디스크 포인터를 따라 다른 페이지를 추가로 읽어야 하는 다중 디스크 조회 부하가 발생합니다.

### 1.4 무의식적인 SELECT * 전송 비용 증가
- 열의 개수가 수십 개가 넘어가는 테이블에 SELECT * 쿼리를 전송하면, 데이터베이스 서버뿐만 아니라 네트워크 전송 대역폭(Bandwidth) 낭비와 클라이언트/애플리케이션 메모리 버퍼에 불필요한 과부하를 가하게 됩니다.

---

## 2. 해결 방안: 수직 분할 (Vertical Partitioning)

가장 대표적인 해결책은 가로로 비대한 테이블을 자주 조회하는 핵심 열 그룹과 부가적인/용량이 큰 열 그룹으로 분리하는 것입니다.

### 2.1 적용 전략
- **핵심 정보와 상세 정보 분리**: 단건 단기 조회용 필드(아이디, 상태코드 등)는 가볍고 빠르게 조인될 수 있도록 날씬한 슈퍼타입 테이블에 두고, 본문이나 상세 설명 등은 별도 서브타입 테이블로 격리합니다.
- **상속 모델링(Subclassing) 활용**: 카테고리나 유형에 따라 쓰이는 속성들이 상이할 때 이를 1:1 관계의 하위 서브 테이블로 나누어 스펙에 맞게 NOT NULL 제약을 걸고 메인 테이블은 작게 유지합니다.

## 3. 실제 카카오톡 DB 스키마 개선 사례 분석

본 프로젝트의 데이터베이스 설계에 반영된 모범 사례들은 다음과 같습니다.

### 3.1 MESSAGES 테이블 상속 모델링
- 변경 전: MESSAGES 테이블에 content(본문 텍스트), S3 파일 상세 정보, 이모티콘 맵핑 코드 등이 한데 모여 있어 텍스트 전송 시 많은 컬럼들이 NULL로 채워졌습니다.
- 변경 후: 공통 메타데이터(id, type, created_at, client_message_token 등)만 MESSAGES 테이블로 들고 가고, 본문 내용은 TEXT_MESSAGES에, 파일 정보는 MEDIA_MESSAGES로 물리적으로 수직 분할했습니다.
- 기대 효과: 대화방의 대량 메시지 렌더링 시 무거운 파일 정보 주입 없이 텍스트 중심의 메시지들만 가볍고 빠르게 조회할 수 있습니다.

### 3.2 PROFILES와 PROFILE_ASSETS 분리
- 변경 전: PROFILES 테이블에 최신 프로필 사진 및 과거 프로필 이력 정보까지 중구난방으로 관리하려 했습니다.
- 변경 후: 현재 쓰이는 최신 프로필 이미지만 PROFILES 테이블에 캐싱해두고, 과거 사진첩 히스토리 조회는 독립된 PROFILE_ASSETS 테이블로 이력을 오프로딩했습니다.
- 기대 효과: 친구 목록 로딩 등 최신 프로필 정보만 대량으로 매핑해야 할 때 불필요한 과거 이력 데이터 스캔 없이 즉각 조회가 가능합니다.

## 4. 개발 가이드라인 및 베스트 프랙티스

- **필요한 컬럼만 명시하여 조회**: 쿼리 작성 시 가급적 SELECT * 사용을 지양하고, SELECT id, nickname 과 같이 필수 필요한 컬럼만을 엄격하게 지정하여 호출합니다.
- **VARCHAR 최대 크기의 합리적 지정**: 사용하지도 않는 varchar(4000) 등을 남발하여 선언하기보다는, 적절한 길이(예: varchar(50))를 부여하여 예측 가능한 메모리 구조를 가질 수 있도록 통제해야 합니다.
- **적절한 인덱스(B-Tree) 설계**: 복합 인덱스 구성 시, 선두 컬럼에는 항상 카디널리티가 높은 핵심 필수 식별자를 선언하고 쿼리에 자주 등장하는 경량 컬럼 위주로 태워 인덱스 파일 크기 자체도 가볍게 유지합니다.
