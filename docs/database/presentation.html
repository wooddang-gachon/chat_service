Part 1. 서론

단순히 글만 적어두는
보관소 설계의 한계

단순한 보관소는 단지 대화 글자들만 대강 적어두는 형태로 시작합니다.
하지만 실제 서비스에서는 예상치 못한 현실적인 문제들이 튀어나옵니다.

🚦 사용자 구분: 로그인 상태와 비로그인 상태는 어떻게 나눌 것인가?
📊 행동 분석: 마케팅 부서에서 요청하는 초기 사용자 행동은 어디에 기록할 것인가?
🔧 기기 고장: 폰이 고장 나거나 바뀔 때 과거 데이터를 안전하게 옮길 방법은 있는가?

이러한 문제를 해결하기 위해, 단순한 구조를 넘어선 사고력이 깃든 보관소 설계가 필요합니다.

따라서 보관소 설계에는 만약의 고장이나 법적인 의무에 대비한 장치가 들어가야 합니다.

Part 2. 기능 분석

카카오톡의 다섯 가지
소중한 데이터 영역

우리가 자주 쓰는 카카오톡은 어떤 데이터를 주로 보관할까요?

👤 1. 회원과 프로필 (내 정보, 친구 설정, 다르게 보이는 내 모습)
💬 2. 대화방 (1대1 귓속말, 단체방, 익명으로 모이는 오픈채팅방)
✉️ 3. 주고받는 대화 (텍스트, 답장, 지운 대화, 수정한 말)
😀 4. 이모티콘 캐릭터 (구매한 이모티콘, 글자 속 미니 이모티콘)
📁 5. 톡캘린더와 파일 (약속 일정, 사진 및 동영상 만료 기한)

📝 수정된 흔적 남기기:
  · 화면에 "편집됨" 표시를 강제 노출.

설계 시 해결해야 하는
18가지 현실적 문제

단순 동작을 넘어, 회원이 탈퇴할 때, 중복 전송이 일어날 때, 수백 명이 대화를 읽을 때 등 오류가 나기 쉬운 현실적인 설계 과제를 어떻게 논리적으로 풀어나갈지 고민한 결과입니다.

Challenge 1

단 한 대의 기기에서만 접속 보장

기존 접속 단말기

📱 로그인 상태가 정상적으로 유지되고 접속 증명서 발급 상태.

🔄 새 단말기 접속이 발생해도 기존 기기의 토큰 만료 처리가 즉각적으로 이루어지지 않으면 중복 접속 허용 사고 발생.

새 단말기 접속 시 대처

⚡ 새 단말기 로그인 즉시 기존 기기의 접속 증명서 값 무효화(수정) 처리.

🔐 중복 등록을 원천 차단하기 위해 유저 아이디 하나당 기기 정보가 단 하나만 등록되게 설계 규칙으로 강제.

⏳ 강제 만료 약속:

Challenge 3

탈퇴 회원 정보 파기와 보존 의무 충돌

법적 충돌 상황

보관소 물리 설계 표준

속도가 늦어지지 않고 정보가 뒤엉키지 않도록 지탱하는 컴퓨터의 물리적 탐색 구조, 테이블 쪼개기 비결, 상속 테이블 관리법을 정리합니다.

📂
Theory 1

가로로 뚱뚱한 테이블의 위험성

⚙️ 페이지 단위 로딩:
  · 컴퓨터는 하드에서 데이터를 한 줄씩 읽는 게 아니라 16KB 크기의 '가상의 상자(페이지)' 단위로 한꺼번에 읽음.
  · 테이블 가로 열(Column)이 많아 한 줄의 크기가 비대해지면 상자 하나에 들어가는 데이터 개수가 확 줄어듦.
  · **결국 조금만 뒤져도 될 일을 수십 배 많은 상자를 읽어야 해서 디스크 속도 지연 초래**.

🛡️ 캐시 효율 저하:
  · 메모리 버퍼 공간 낭비로 캐시 적중률 급락.

42
Theory 2

페이지 넘침과 테이블 나누기

페이지 넘침 (Row Overflow)

⚠️ 외부 이탈:
  · 한 행의 용량이 8KB를 초과하면, 컴퓨터는 용량이 큰 가변 텍스트를 페이지 바깥으로 내쫓아 따로 보관함.

❌ 이중 탐색:
  · 행에는 바깥을 가리키는 주소표만 남아, 글을 읽을 때마다 바깥 주소를 다시 확인하러 가는 이중 조회 부하 유발.

테이블 수직 나누기

⚡ 가벼운 메인 서랍:
  · 상태, 코드 등 가벼운 정보는 메인 서랍에 두고 항상 빠르게 조회.

🛡️ 상세 서랍 격리:
  · 길고 무거운 상세글 정보는 별도 자식 서랍으로 격리하여 메인 조회 속도와 캐시 적중률 완전 보호.

🌳
Theory 3

인덱스 글자 크기와 찾아보기 속도

⚙️ 찾아보기 목록(인덱스)의 한계:
  · 책 뒷장의 '찾아보기' 글자 자체가 너무 길면, 한 페이지에 몇 글자 적지 못함

🚦 탐색 단계 늘어남:
  · 찾아보기 리스트 장수가 많아지므로 단어 하나를 고속 색인하기 위해 거쳐야 하는 책장 탐색 횟수가 늘어남

🔍 권장 방향:
  · 찾아보기 목록에 묶을 컬럼들은 최소한으로 좁히고 가급적 용량이 작은 가벼운 규격으로 구성

Theory 4

순차 정렬과 기본 보관 규칙

무작위 순서 저장 (UUID 등)

❌ 중간 쑤셔넣기:
  · 데이터가 중간중간 빈자리에 무작위로 끼어들어감.

⚠️ 책장 뒤흔들기 (Page Split):
  · 꽉 찬 페이지 중간에 줄을 끼워넣느라 뒷줄 데이터를 전부 밀어내고 쪼개는 디스크 부하 및 락 발생.

순차 번호 저장 (Auto Increment)

⚡ 꼬리 차곡차곡 쌓기:
  · 신규 데이터가 항상 보관 테이블 맨 마지막 페이지 꽁무니 뒤에 이어 붙음.

🛡️ Zero Split:
  · 책장 순서를 뒤흔들 일이 전혀 없어 컴퓨터에 가해지는 쓰기 압박이 제로화.

Theory 6

전체 다 뒤지기 방지와 구간 스캔

🚫 **처음부터 끝까지 다 뒤지기 (Full Scan)**:
  · 조건 없이 검색을 돌리면 컴퓨터가 1억 개의 대화를 1번부터 전부 다 읽어야 해서 시스템 정지 발생.

🛡️ **가상의 칸막이 치기 (Index Range Scan)**:
  · 대화방 번호(`chat_room_id`) 조건 필터를 걸어 특정 방 데이터가 모여 있는 구역만 콕 집어 점프 검색.

📖 **한 번에 한 장씩 보여주기 (Paging & Limit)**:
  · 사용자가 화면을 위로 올릴 때만 `최근 20개`씩 한정 끊어 읽어 들여 컴퓨터 리소스 낭비를 원천 봉쇄.

데이터 영구 삭제와 휴지통 보관

휴지통에 보관 (Soft Delete)

📂 흔적 남기기:
  · 회원이 나갔거나 취소한 행위라도 데이터로서 가치가 있는 경우.
  · 지움 도장(Flag) 또는 지운 일시만 기록해두고 보존.

💡 **가치 분석**: 디스크 저장 비용은 매우 싸기 때문에 비즈니스 분석 가치를 지키기 위해 적극 추천.

진짜 영구 삭제 (Hard Delete)

🗑️ **즉시 소멸**:
  · 두 정보의 연결 고리만을 단순히 매핑하는 중간 연결 테이블 정리 시.

⚡ **단순한 구조**: 불필요해진 껍데기 연결 정보는 하드웨어 디스크에서 물리 삭제해 테이블 상태 유지.

DDL 1

회원 및 디바이스 스키마 DDL

-- 1:1 수직 분할: user와 profile을 쪼개어 캐시 효율 극대화
CREATE TABLE `user` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `phone_number` VARCHAR(50) NOT NULL UNIQUE,
    `email` VARCHAR(100) UNIQUE,
    `status` VARCHAR(20) NOT NULL DEFAULT 'ACTIVE'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE `profile` (
    `user_id` BIGINT PRIMARY KEY,
    `nickname` VARCHAR(50) NOT NULL,
    `profile_image_url` VARCHAR(255),
    FOREIGN KEY (`user_id`) REFERENCES `user` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

DDL 2

친구 및 멀티프로필 DDL

-- friendship 테이블: 빠른 조회를 위한 B-Tree 복합 색인 추가
CREATE TABLE `friendship` (
    `user_id` BIGINT NOT NULL,
    `friend_user_id` BIGINT NOT NULL,
    `status` VARCHAR(20) NOT NULL DEFAULT 'NORMAL',
    `is_favorite` TINYINT(1) NOT NULL DEFAULT 0,
    PRIMARY KEY (`user_id`, `friend_user_id`),
    FOREIGN KEY (`user_id`) REFERENCES `user` (`id`) ON DELETE CASCADE,
    FOREIGN KEY (`friend_user_id`) REFERENCES `user` (`id`) ON DELETE CASCADE,
    -- 찾아보기용 복합 인덱스 지정
    INDEX `idx_friend_user` (`friend_user_id`, `user_id`),
    INDEX `idx_user_friend_favorite` (`user_id`, `status`, `is_favorite` DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

    `id` BIGINT AUTO_INCREMENT,
    `type` VARCHAR(20) NOT NULL,
    PRIMARY KEY (`id`, `type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE `chat_room_member` (
    `chat_room_id` BIGINT NOT NULL,
    `user_id` BIGINT NOT NULL,
    `joined_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    `leaved_at` TIMESTAMP NULL,
    `cleared_at` TIMESTAMP NULL,
    PRIMARY KEY (`chat_room_id`, `user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
