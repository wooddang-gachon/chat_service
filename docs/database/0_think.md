# 서론

기존 db는 채팅서비스에 대한 최소기능만 정의한 테이블이다. 물론 스터디 측면에서는 가능하지만 이게 진정한 백엔드 스터디라 할 수 있을까?

단순한 문법에 대한 내용은 이미 ai를 통해 작업이 가능하다. 우리는 이제 ai를 이용한 빠른 문제 접근과 이해, 그 중에 발생가능한 tradeoff를 결정하는 **사고력**이 중요해진 시기이다. 나는 이 부분을 테스트 겸 이렇게 기록하면서 진행 예정이다.

# 본론

## 문제인식(우리의 진정한 목표가 무엇인가?)

### 우리가 설계한 테이블은 어떤 고려가 되어있나?

저번주 db 때 선규(db설계를 좀 해본 사람)이 하는 질문이 몇가지 있었다.

- user table 하나만 만들면 이 유저를 구분할 일(로그인, 비로그인, 게스트)이 있을 때 어떻게 분기를 할 것이냐
- 영업 측면에서 특정 사이트를 처음 방문한 경우 유저가 남아있는 1분간의 행동에 대한 로그가 중요한데 이걸 어떻게 처리할 것인가?

이러한 질문에 테이블 3개 밖에 없는 나는 쉽게 대답을 하지 못했다. 사실 최소 기능만 만족하는 테이블이기 때문에 이는 당연하다. 그럼 진정으로 db설계 실력을 기르기 위해서는 어떻게 해야될까?

### 백엔드개발자스러운 db설계는 무엇일까?

사실 이 부분에 대한 답은 선규를 통해 들을 수 있었다. db는 최소 기능을 만족하는게 아니라 해당 서비스에서 발생할 수 있는 모든 케이스에 대한 고민이 들어간 설계를 말하는 것이다.

해당 서비스가 단순히 동작을 하기 위한 것이 아니라, 영업에서 필요한 데이터, 이후 개발계획까지 고려하고, 데이터에 이상이 발생할 경우에 대한 예비책 또한 db테이블 설계에 고민의 흔적이 들어가야 한다.

### db는 서비스 이전에 회사의 모든 자산이 정리되는 공간

사실 db를 설계할 때 핵심적인 관점은 이런 관점이 아닐까? 그럼 내 목표는 회사 내에 발생할 수 있는 모든 케이스를 고려해서 db를 만들어야되지 않을까.

## 문제 정의

우리는 카카오톡 서비스를 보고 db를 설계하기로 숙제를 정했다. 하지만 이건 최소 기능만을 구현한 우리가 실제 서비스를 클론하며 위의 현실적인 질문들을 던져볼 수 있게 만든 **최소 요구**사항이었다.

그렇다면 나는 무엇을 해야할까?(리더로서 모범이 될만한 접근법)

### 리더로서

역시 최소 요구사항을 이행하면서 방식은 다르게 접근하며, 최고의 결과물을 내는 것이 가장 이상적인 형태다.

결론은db 설계를 고려하는 나름의 ai 에이전트를 만들어보는 것이다.

## 문제 해결 방법

어떤 문제든 방식은 똑같다.

1. 요구 문제에 대한 가장 근원적인 요구를 정의한다.
2. 문제를 해결과정에 대한 로드맵을 머릿 속에서 그린다.
3. 로드맵 과정에 필요한 지식을 최대한 많이 쓸어담는다.
4. 쓸어담은 지식을 공학적으로 해결 가능한지 판단한다.
5. 될 때까지 반복한다.
