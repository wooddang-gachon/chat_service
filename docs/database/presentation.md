# 카카오톡 데이터 보관소 설계 및 최적화 전략 발표자료 (presentation.md)

이 문서는 카카오톡의 정보 보관소(데이터베이스) 설계 시 직면하는 현실적인 비즈니스 고민과 해결 방안을 비기술자도 한눈에 이해할 수 있도록 쉬운 일상 언어로 정돈한 72페이지 분량의 발표자료 구성안입니다.

---

# 슬라이드 1: 표지
- **슬라이드 제목**: 카카오톡 데이터 보관소 설계 고민과 최적화 전략
- **서브 타이틀**: 서비스가 멈추지 않고, 정보가 뒤엉키지 않게 데이터를 보관하는 방법
- **작성/발표**: 데이터 설계 및 보관소 최적화팀
- **일시**: 2026년 7월 16일
- **스피커 노트**: "안녕하세요. 오늘 발표할 주제는 카카오톡 데이터 보관소 설계의 핵심적인 고민과 그 해결 방법입니다. 어려운 기술 용어를 최대한 걷어내고, 누구나 이해할 수 있는 개념 위주로 친근하게 설명드리겠습니다."

# 슬라이드 3: [Part 1] 단순히 글만 적어두는 보관소 설계의 한계
- **주제**: "우리가 설계한 데이터 서랍에는 어떤 고민이 담겨 있는가?"
- **문제 인식**: 
  - 단순히 대화방 서랍 몇 개만 만들어 글자만 모아두는 것은 실제 서비스에서 한계에 부딪힙니다.
  - 로그인 상태와 게스트 상태 구분, 마케팅 부서에서 원하는 초기 사용자 스크롤 분석, 핸드폰 고장 시 데이터 이전 등의 현실 문제를 풀기 어렵습니다.
- **핵심 메시지**: 정보 보관소는 단순 기계가 아니라 서비스에서 일어날 수 있는 모든 예외 상황을 품는 비즈니스의 자산입니다.

# 슬라이드 4: [Part 1] 데이터 설계의 진짜 의미
- **주제**: 비즈니스 흐름과 안전망을 함께 보관하는 서랍
- **핵심 요소**:
  - **영업적 가치**: 유입 채널 분석 등 마케팅을 위한 데이터 구조 선제 마련
  - **미래 확장성**: N:M(다대다) 관계와 기능 변화를 미리 대비하기
  - **장애 대비책**: 비정상적인 유입이나 예외 속에서도 정보가 뒤엉키지 않게 보호하기
- **스피커 노트**: "단순히 돌아가게만 만드는 것이 아니라, 회사의 자산이 안전하게 정돈되고 만약의 상황에서도 작동을 보장하는 보관 장벽을 세우는 것이 진정한 설계입니다."

- **보관소 설계**:
  - 백업 메타 데이터 서랍에 만료 날짜(`expires_at`)를 지정하고, 매일 도래한 만료일을 자동 대조하여 파일과 메타 정보를 깔끔하게 삭제

# 슬라이드 36: [Part 3] [과제 16] 채팅방 나가기와 대화 삭제 시의 뷰 격리

- **보관소 설계**:
  - 대화 글은 하나의 테이블에 한 번만 저장
  - 멤버 테이블에 각자 나간 시각(`leaved_at`) 및 대화 비운 시각(`cleared_at`)만 보존. 대화 로딩 시 `대화시작시각 > 비운시각` 필터링으로 물리 복제 없이 격리 뷰 보장

# 슬라이드 37: [Part 3] [과제 17] 대용량 파일 자동 파기 스케줄 관리

- **해결 과제**:

  - 업로드 시점에 유저의 구독 요건 상태를 확인해 일반인은 `NOW() + 14일`, 구독자는 `Null`로 설정. 보관 스케줄러가 만료일 지난 미디어를 수거 하드 삭제

# 슬라이드 39: [Part 4] 가로로 뚱뚱한 테이블의 위험성
- **주제**: 가로 컬럼 개수가 너무 많아질 때 발생하는 성능 저하 원인
- **속도 저하 원인**:
  - **가상의 상자(페이지) 단위 로드**: 컴퓨터는 16KB 크기의 상자 단위로 하드에서 메모리로 데이터를 한 번에 가져옴. 가로 열(Column)이 비대하면 상자 하나에 데이터가 몇 개 안 들어가서 동일 건수 조회 시 훨씬 많은 상자를 열어야 함
  - **메모리 캐시 효율 급감**: 페이지당 데이터 밀도가 낮으면 메모리가 비효율적으로 낭비되어 캐시 적중률이 하락함

---

# 슬라이드 40: [Part 4] 페이지 넘침과 테이블 나누기
- **주제**: 로우 오버플로우와 수직 분할의 개념
- **대비 설계**:
  - **페이지 넘침(Row Overflow) 차단**: 한 줄 용량이 8KB를 초과해 가변 텍스트 컬럼이 외부 페이지로 이탈, 조회 시 외부 디스크 포인터를 추가 탐색하느라 이중 조회 지연이 발생하는 현상 방지
  - **테이블 수직 나누기**: 자주 쓰는 가벼운 정보는 메인 테이블에 두고, 무겁고 큰 상세 텍스트는 1:1 관계의 자식 테이블로 격리 보존

---

# 슬라이드 41: [Part 4] 인덱스 글자 크기와 찾아보기 속도
- **주제**: 인덱스 키 바이트 크기가 B-Tree 깊이에 미치는 영향
- **색인 작동 원리**:

  - 이로 인해 찾아보기 목록 책장 깊이가 늘어나, 단어를 고속 색인하기 위해 <strong>루트 노드부터 거쳐가야 하는 책장 탐색 횟수가 증가해 조회 속도 저하</strong>

---

# 슬라이드 42: [Part 4] 순차 정렬과 기본 보관 규칙
- **주제**: 기본키(PK) 물리 정렬과 페이지 분할 방지
- **물리적 정렬**:
  - MySQL InnoDB 엔진은 데이터 저장 시 기본키(PK)를 기준으로 물리적인 정렬 순서를 유지함 (Clustered Index)
  - PK로 무작위 값(UUID 등)을 사용하면 데이터 삽입 시 기존 정렬된 데이터의 중간 임의 위치에 삽입되어, 꽉 찬 페이지를 쪼개는 <strong>페이지 분할(Page Split)이 빈번해지고 디스크 쓰기 압박 유발</strong>
  - **해결**: PK는 항상 순차적으로 증가하는 Auto Increment를 채택

---

# 슬라이드 43: [Part 4] 찾아보기 목록 개수와 글쓰기 속도 (RDB 튜닝)
- **주제**: 인덱스 추가에 따른 쓰기 성능 저하와 최적의 절충안
- **핵심 내용**:
  - 원하는 조건의 색인 목록이 적재되어 있으면 책장을 몇 장 넘기지 않고도 원하는 데이터를 즉각 획득 (읽기 초고속화)
  - 하지만 새 글이 들어올 때마다 뒷장 찾아보기 목록(인덱스) 위치도 일일이 다시 찾아 정렬해 적어야 하므로 **쓰기(INSERT/UPDATE) 연산 락 지연 발생**
  - **해결**: 자주 쓰이는 조건에만 최소한의 결합 색인을 쥐어주고 쓸데없는 중복 인덱스 생성을 철저히 지양하는 설계 필요

# 슬라이드 44: [Part 4] 전체 다 뒤지기 방지와 구간 스캔
- **주제**: 쿼리 스캔 최소화 설계와 범위 검색
- **최적화 기법**:
  - 조건 없이 검색을 돌리면 컴퓨터가 1억 개의 대화를 1번부터 전부 다 읽어야 해서 시스템 정지 발생 (Full Scan)
  - 대화방 번호(`chat_room_id`) 조건 필터를 걸어 <strong>특정 방 데이터가 모여 있는 구역만 콕 집어 점프 검색 (Index Range Scan)</strong>
  - 한 번에 한 장씩 보여주는 페이징(`LIMIT 20`) 처리를 결합해 컴퓨터 디스크 낭비를 원천 봉쇄

- **주제**: 발표 마무리 및 질의응답
- **예상 질의와 핵심 답변**:
  - Q: NoSQL 대신 관계형 데이터베이스(MySQL)를 사용한 이유는 무엇인가요?
    - A: 돈이 오가는 기프티콘 거래와 회원 정산 원장의 데이터 일관성이 절대적으로 중요하며, 테이블 간 강력한 제약과 안전한 관계 보장이 필요하기 때문입니다.
  - Q: 1:1 수직 분할로 인해 오히려 테이블을 결합(JOIN)할 때 성능이 떨어지지 않나요?
    - A: 로그인이나 접속 세션 갱신 등 계정 정보만 가볍게 읽는 빈도가 프로필 전체 상세 조회를 읽는 빈도보다 수십 배 높기 때문에, 핵심 스캔 부하를 낮추는 이득이 훨씬 큽니다.
